本地事务：本服务事务实现
常见应用：Spring 的 @Transactional
原理：基于AOP + JDBC 事务规范
实现：事务AOP拦截器触发，获取事务管理器，创建事务上下文（ThreadLocal保证方法内的SQL使用同一个连接），获取连接， 设置autoCommit关闭， 统一提交或回滚（连接的方法）
适用：基于 ACID 模型、支持 XA/JTA 协议的关系型数据库， 并且是单一数据库连接（同一个事务上下文）
其他：对于设计noSQl或者其他库连接操作的情形无法生效， 需要手动编写正向操作+反向补偿的方法保证；
事务传播： 对于多个事务方法之间的调用， 如何管理事务上下文的策略， 是新建、复用还是放弃？
包括：（1）REQUIRED 默认， 当前有事务则加入，否则新建 （2）REQUIRES_NEW， 无论有无都新建事务， 不影响主事务如日志服务；
     （3）SUPPORTS 有事务加入， 没有不创建， 如一些只读操作（4）NOT_SUPPORTED ，不管有没有事务，都不用事务（原有事务暂停），耗时的非核心操作
     （4）NEVER 不允许有事务，有则抛异常， 绝对制度不允许修改的场景
事务隔离级别：（1）READ_UNCOMMITTED 读未提交；脏读、不可重复读、幻读都可能出现， 几乎不用；写操作加行级排他锁（X 锁），但事务提交前不释放， 读操作不加锁，直接读取最新的物理数据（不管是否提交）
            （2）READ_COMMITTED 读已提交，常用；会出现不可重复读， 写操作：加行级 X 锁，事务提交后释放；基于MVCC，每次读操作生成一个 “快照”（Read View），只可见已提交的版本；每次 SELECT 都会生成新的 Read View → 同一个事务内多次读同一行，可能读到不同结果
            （3）REPEATABLE_READ 可重复读， 默认；仍然会出现幻读；写操作：加行级 X 锁 + 间隙锁（Gap Lock）（防止幻读），事务提交后释放；读操作：基于 MVCC，整个事务内只生成一次 Read View → 同一个事务内多次读，都用同一个快照，结果一致；间隙锁：锁定行之间的 “间隙”（比如 WHERE id > 10），防止其他事务插入数据，解决大部分幻读问题
            （4）SERIALIZABLE 事务串行无并发， 写操作：加行级 X 锁 + 表级锁；读操作：加行级共享锁（S 锁），直到事务提交才释放；
配置：隔离级别只对写事务 / 读写事务生效，纯只读事务（readOnly = true）的隔离级别配置会被忽略
注意： 事务重点是本事务中的操作整体性提交或回滚， 事务的不同隔离级别关注的是数据“读 - 写冲突”， 不能精准解决针对数据行的并发“写-写冲突”，所以为解决并发问题还是需要在数据操作时使用诸如for update 之类的加锁限制；
乐观锁： 可以用 UPDATE 的 WHERE 条件替代 for update 的锁（X排他锁），靠数据库的行级锁原子性执行
混淆： update或者delete 引擎会自动加写锁， 作用时保证写操作的原子性， 不会互相干扰没有业务含义；而 select for update的加锁时再读时就加锁。 事务提交才释放， 所以可以保证先查后改场景的并发问题；
区分：悲观锁靠 “提前锁” 保证成功率，牺牲性能；乐观锁靠 “瞬时锁” 提升性能，接受一定的失败率。
        锁类型	加锁时机	锁持有时间	性能	失败场景	失败处理方式
        悲观锁（FOR UPDATE）	SELECT 时加锁	整个事务周期（查→改→提交）	低（锁等待多）	几乎无（提前锁定，不会超卖）	极少失败，仅需处理数据库异常
        乐观锁（带条件 UPDATE）	UPDATE 时加锁	仅 UPDATE 执行的瞬间	高（无锁等待）	库存不足（UPDATE 影响行数 = 0）	必须处理失败，返回提示 / 重试
乐观锁类型： 版本号（版本一致则写）、条件判断（比如数量条件、状态条件等，满足要求则写）、幂等键（如Mysql的 INSERT ... ON DUPLICATE KEY UPDATE，存在更新不存在插入的原子执行）、redis乐观锁（原子增加increment）
死锁分析：SELECT FOR UPDATE id = *** 查询不存在的记录， 不会加行锁，而是加间隙锁（Gap Lock），高并发下，多个事务争抢重叠的间隙锁，且加锁顺序不一致 → 形成循环等待 → 触发死锁。
        解决方法（1）降隔离级别为读提交不会加间隙锁（2）使用INSERT ... ON DUPLICATE KEY UPDATE无需加锁 （3）控制加锁顺序， 比如按序号最小先加锁而不是随机加锁导致区间重叠;
